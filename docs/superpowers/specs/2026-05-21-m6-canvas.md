**Parent spec:** `docs/superpowers/specs/2026-05-16-water-design.md` (§ 4.8 M6 plan).

## 1. One-paragraph product context

M1–M4 gave Water the writing surface, pill engine, character bibles, and world bibles. M5 gave the writer a per-scene barometer — heat that they can read at a glance while writing. M6 zooms out: a **macro spatial canvas** where the whole project lays out as movable cards, each carrying a tiny heat sparkline. The writer arranges scenes in space (act layouts, parallel storylines, geographic clusters) without changing manuscript order. They drag, they group, they see at a glance which scenes carry which heat — where the cold beats are, where the cast crowds, where pacing thins out. It's where the writer steps back from the page to see the shape of the thing.

The canvas is observational, not generative. No LLM, no pills on this surface. Pure spatial introspection.

---

## 2. Design intent (the un-violable rules)

All M1–M5 hard constraints carry forward. M6 specifics:

1. **Canvas position is private writer-tooling, not narrative position.** Manuscript ordering remains authoritative for what the reader reads. Canvas position is the writer's mental map.
2. **Local-first.** Position writes hit SQLite + scene-md frontmatter (so manual git-tracked moves survive). No cloud sync.
3. **Smooth pan + zoom.** This is the only surface where the writer reads-by-eye rather than typing. It must feel like a real canvas, not a paginated grid.
4. **Heat sparklines are the same data as the strip.** No new compute path — sparklines pull from `HeatStore` and downsample. Toggling a metric on the canvas surface uses the same `heat_set_metric_enabled` setting as the editor strip.
5. **Pastel-glow visual identity** carries through. Scene cards live on the cool-gray substrate with the same elevation tokens as character/world tiles.

A new fourth nav target on the IconRail:


Icon: a small grid-of-dots glyph (Lucide `LayoutGrid` or `Map` — sketch in the implementation plan).

Clicking it routes to the new `<CanvasSurface />`. The canvas takes the full editor area; pan-zoom interactions live within it.

## 4. Interaction model

| Gesture | Effect |
|---|---|
| **Drag empty space** | Pan the canvas. |
| **Cmd/Ctrl + scroll** | Zoom in/out, centered on cursor. Min zoom = fit-all-scenes; max zoom = 2×. |
| **Click a scene card** | Open that scene in the editor (route to `scenes` nav, activeSceneId = clicked id). |
| **Drag a scene card** | Move the card. Position persists via debounced `scene_canvas_set_position`. |
| **Shift-drag a scene card** | Select rectangle — drag-out a marquee, all overlapped scenes join a group. |
| **Cmd/Ctrl-click multiple cards** | Multi-select for batch-move. |
| **Right-click empty space** | "Reset to manuscript order" action — re-flows every card into a left-to-right row using manuscript ordering. Confirms with a dialog because it discards spatial work. |
| **Top-right metric chip** | Same picker primitive as the editor strip; chooses which metric the sparklines render. Default: Pacing. |
| **Reading-order overlay toggle (top-left)** | When on, draws thin lines between scenes in manuscript order. Useful for diagnosing "did my spatial layout match my narrative order?" |

Discoverability: on first open with no `canvas_x`/`canvas_y` set yet, surface a one-time tooltip pointing at empty space: *"drag scenes anywhere. cmd-scroll zooms."* with × dismissal (localStorage-persisted, mirrors the M5 intro pattern).

## 5. Scene cards + heat sparklines

Each scene card is approximately 200 × 100 px at zoom = 1.

```
┌──────────────────────────┐
│  The Library's Secret    │  ← serif title
│  ────                    │  ← hairline
│  ▒░▓▒░▒░ █ ░             │  ← heat sparkline (60 × 8 px)
│  pacing                  │  ← active metric name
│                          │
│  734 words · 8 paragraphs│  ← muted footer
└──────────────────────────┘
```

- **Background:** `var(--water-bg-raised)` with `var(--water-elev-1)` shadow. Hover bumps to `--water-elev-2` and lifts 2 px (same as world/character tiles).
- **Sparkline:** reuses an extracted `HeatmapStripView` presentational component (Task 4 of the implementation plan). At small sizes it renders fewer columns — one column per paragraph; if there are > 60 paragraphs, downsample by averaging adjacent paragraphs (compute happens in the renderer; the data is already small).
- **Metric color:** matches the editor strip's hue mapping for the active metric.
- **Hover:** the card grows a small phrase tooltip (same phrasebank) summarizing the metric's dominant shape ("slow burn", "thick cast", etc. averaged over the scene).

## 6. Engine architecture (sketch)

### 6.1 New SQL schema (v6)

```sql
ALTER TABLE scene ADD COLUMN canvas_x REAL;
ALTER TABLE scene ADD COLUMN canvas_y REAL;
ALTER TABLE scene ADD COLUMN canvas_group TEXT;  -- nullable, v1 group label

UPDATE schema_version SET version = 6;
```

`canvas_x` / `canvas_y` are nullable so existing scenes "default" to unplaced; the renderer auto-flows them into a row by manuscript order on first open.

### 6.2 Rust surface

- `crates/water-core/src/scene.rs` — `SceneStore::set_canvas_position(scene_id, x, y)` and `set_canvas_group(scene_id, group_or_none)`. Also extend `SceneFile` (frontmatter) so the values round-trip to disk (writer-readable, git-trackable).
- `crates/water-core/src/canvas/auto_flow.rs` — `auto_flow(scenes: &[SceneRow]) -> Vec<(Id, f32, f32)>` — deterministic initial layout in manuscript order.

### 6.3 Tauri commands

- `scene_canvas_list() -> Vec<SceneCanvasRow>` — returns id, name, manuscript_order, canvas_x, canvas_y, group.
- `scene_canvas_set_position(scene_id, x, y)` — debounced from the renderer.
- `scene_canvas_set_group(scene_id, group_or_null)`.
- `scene_canvas_reset_all()` — used by the "reset to manuscript order" right-click action.

### 6.4 Renderer

- `app/src/canvas/CanvasSurface.tsx` — the new top-level view (pan/zoom + scene cards + reading-order overlay).
- `app/src/canvas/SceneCard.tsx` — one card. Receives scene + sparkline data as props.
- `app/src/heat/HeatmapStripView.tsx` (extracted from `HeatmapStrip`) — pure visual; takes metrics + active kinds. Reused both by the editor strip and the canvas sparklines.
- `app/src/canvas/CanvasIntro.tsx` — first-open tooltip.
- `app/src/styles/canvas.css` — pan/zoom container + scene-card styling.

## 7. Exit criteria

- [ ] Canvas surface mounts on the new "canvas" IconRail target.
- [ ] All scenes in the active manuscript render at their persisted positions (or at the auto-flow position if unplaced).
- [ ] Each card shows a 60 × 8 sparkline of the active metric, hue-matched to the metric.
- [ ] Dragging a card persists position via debounced IPC and the next reload shows it in the same spot.
- [ ] Pan + Cmd-zoom both work smoothly (no jank on a 30-scene project).
- [ ] Click a card → routes to scene nav with that scene active.
- [ ] Reading-order overlay toggles with a single keystroke (`O`?) and draws thin lines between scenes in manuscript order.
- [ ] "Reset to manuscript order" right-click + confirm restores auto-flow positions.
- [ ] Metric chip works (changes which metric the sparklines render).
- [ ] First-open intro tooltip appears once, dismisses on ×, persists.
- [ ] `cargo test -p water-core canvas::` is green.
- [ ] Manual smoke walk against an M6 acceptance fixture (a 6-scene project with a clear spatial layout — three "Act I" beats clustered, three "Act II" beats clustered) lands the expected canvas state.

## 8. Explicitly NOT in M6

- Multi-manuscript canvases (separate canvas per manuscript book, etc.).
- Auto-layout heuristics ("snap to act boundaries", "arrange by sentiment cluster").
- Macro-level heat aggregation (a "project-wide heatmap" overlaying the canvas as a heightmap) — M7+ candidate.
- Collaborative pointers / shared canvas state — out of scope; Water is single-user.
- Touch gestures (pinch zoom, two-finger pan) — desktop-first; mouse-first design. If the platform exposes them via PointerEvent the canvas may incidentally support them, but no testing.
- Group as a model entity — v1 group is a string label on the scene row, not its own table. Groups don't have descriptions, manuscripts, dates, etc.

## 9. Open questions — resolutions

The five open questions resolved at lock time:

1. **Canvas IconRail icon** — Lock: `Map` (Lucide). Pairs with the world `Globe` in the same geographic-register family; the writer reads the canvas as a map of their project.
2. **Default zoom** — Lock: **fit-all-scenes** on first open with > 3 scenes; **1.0** when ≤ 3. Heuristic switch lets new projects feel comfortably-sized while big projects still get the overview.
3. **Reading-order overlay style** — Lock: **smoothed curves with subtle endpoints**. Cubic Bezier with control points pulled inward, ~30% opacity. Avoids criss-cross gore when spatial layout diverges from manuscript order.
4. **Card title typography** — Lock: **title-serif (20 px)**. Consistent with the rest of Water's display hierarchy without being heavy at card-size.
5. **Initial auto-flow direction** — Lock: **left-to-right rows**, wrapping at viewport-width-minus-padding. Most natural for English manuscript reading order. 8-card row width, with row gap matching the gap-token rhythm elsewhere in the app.
