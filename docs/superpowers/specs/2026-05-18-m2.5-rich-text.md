## 1. Background

M2's block editor (T8–T10, commit `43b7826` and follow-ups) ships ProseMirror with paragraph / heading / scene_break / dialogue / ordered_list / bullet_list blocks. Block IDs (`^bk-XXXX`) are preserved across all transactions. The serializer is line-oriented and **intentionally lossy for inline marks** — the implementer flagged it in their T8 report (*"M2 doesn't ship bold/italic/links in the block editor; M3+ may"*).

During M2 smoke testing the writer flagged this gap. Character voice prompts in M3 will want italicized internal thought; pinned-pill detail sheets may want bolded speaker names; revision-pass workflows want italic emphasis. The omission blocks no M3 feature, but it scrapes against the writing experience M2 ships.

M2.5 closes that gap with a small, focused patch: three inline marks (bold, italic, link), CommonMark round-trip on disk, keyboard shortcuts + pastel-glow selection toolbar, and a Mod-click-to-open-in-browser affordance.

It is not a milestone. It is a `.5` patch on top of `m2`, modeled on `m1.5.1`.

---

## 2. Goals & non-goals

### 2.1 Goals

1. **Three inline marks** — `strong` (bold), `em` (italic), `link` (href). Compose freely.
2. **CommonMark round-trip** — `**bold**`, `*italic*`, `[text](url)` in the on-disk `.md`. Block IDs (`^bk-XXXX`) continue to anchor at the start of each line.
3. **Keyboard shortcuts** — `Mod-B`, `Mod-I`, `Mod-K`.
4. **Selection toolbar** — pastel-glow capsule above non-collapsed selection, showing B/I/Link icons with active-mark states.
5. **Link popup** — URL input + Apply + Remove (when editing). XSS guard against `javascript:` / `data:` URLs.
6. **Mod-click opens external** — `tauri-plugin-shell` integration; rendered links open in the user's default browser on Cmd-click (Mac) / Ctrl-click (Win/Linux).

### 2.2 Non-goals (explicit)

- Strike, code, underline marks. Cheap follow-ups if writer demand surfaces; not in M2.5.
- Markdown input rules (`**word**` autoconvert as you type). Power-user feature; deferred to M7 settings opt-in.
- Marks inside pill text (the orchestrator emits one-line prose per spec § 6.3; the LLM can include markdown delimiters in its output but the pill UI doesn't render them in M2.5).
- Marks in the scene title `<input>`. Titles stay plain text.
- Markdown input rule for headings (typing `## ` to convert to h2). Deferred.
- KNOWN_FRAGILE #11 (single-shot LLM rate-limiter bypass). Separate `m2.0.1` patch.
- New KNOWN_FRAGILE entries from M2 milestone review (#8 anti-loop English-only, #9 anchor stability, #10 inflection scan). Same priority as KF #11; addressed later.

## 3. Hard constraints (inherited from master spec § 1.2 and § 5.2)

1. **Human-readable on disk.** CommonMark syntax. Any text editor opens an exported `.md` and recognizes the formatting.
2. **Pastel-glow visual identity.** Selection toolbar matches the M2 pill design language. No 1 px borders. No shadows under 6 px blur. Glow signals meaning.
3. **Notion/Apple-fluid motion.** Toolbar fade-in 240 ms (`--water-dur-small`) with `--water-ease-out-soft`. Fade-out 120 ms (`--water-dur-tiny`) with `--water-ease-in-out-water`. Reduced-motion → 1 ms via existing tokens.
4. **Plugin seam friendliness.** The serializer + parser surface is structured so additional marks land as one-line additions per mark, not as schema rewrites.
5. **Determinism on round-trip.** A doc serialized to markdown then parsed back returns an identical document (modulo mark-encoding choices like `**` vs `__` — we pick `*`/`**` canonically). Property test enforces.
6. **`forbid(unsafe_code)`** at the Rust crate root — N/A for M2.5 (renderer only).

## 5. Schema additions (`app/src/editor/schema.ts`)

The Water schema currently extends `prosemirror-schema-basic` with custom nodes and a `blockId` attr on every block. The basic schema already defines `strong`, `em`, and `link` mark specs. M2.5 adds them to the Water schema's `marks` set:

```ts
import { Schema } from "prosemirror-model";
import { schema as basicSchema } from "prosemirror-schema-basic";

// Use the basic schema's mark specs directly — they are CommonMark-compatible:
//   strong: parseDOM `[{ tag: "strong" }, { tag: "b" }]`, toDOM `["strong", 0]`
//   em:     parseDOM `[{ tag: "em" }, { tag: "i" }]`, toDOM `["em", 0]`
//   link:   parseDOM `[{ tag: "a[href]", getAttrs }]`, toDOM `["a", { href }, 0]`,
//           with attrs `{ href: { default: null }, title: { default: null } }`
const marks = basicSchema.spec.marks;  // strong + em + link + (code, also present but unused)

export const schema = new Schema({ nodes: withLists, marks });
```

The basic schema's `code` mark is exposed by importing but unused; it has no keyboard shortcut, no toolbar button, and no serializer rule. Writers who type backticks get a backtick character (no auto-convert). Deferred non-goal.

`link.attrs.title` is preserved for HTML-paste round-trip but never set by Water's serializer (CommonMark `[text](url)` doesn't carry a title; the `[text](url "title")` variant is rare and not supported in M2.5).

## 6. Serializer round-trip (`app/src/editor/serialize.ts`)

### 6.1 Serializer extension

The existing `markdownFromDoc` iterates top-level blocks and emits one line per block via `node.textContent` (which strips marks). M2.5 replaces the per-block `textContent` call with an inline-walking helper:

```ts
function inlineToMarkdown(node: PMNode): string {
  // Walks node's inline content via node.forEach, tracking open marks.
  // For each text run:
  //   1. Diff its marks against the currently-open-marks stack:
  //        - Emit close delimiters for marks that were open but aren't on this run.
  //        - Emit open delimiters for marks on this run that weren't open.
  //   2. Emit the text run's text, with literal `*` `_` `[` `]` `(` `)` `\` escaped via `\`.
  // After the last run, emit close delimiters for any still-open marks.
  // Link marks consume their own text + url; they're written as `[textContent](url)`
  // rather than via open/close delimiters since CommonMark links are atomic.
}
```

Output rules:
- `strong` → `**...**`
- `em` → `*...*` (canonical; `_..._` is read but not written, to avoid ambiguity with intra-word emphasis)
- `link` → `[text](url)`, where text is the run's marked text-content (escaped) and `url` is the href attr
- Nested marks emit in a fixed order: link wraps the outermost, then strong, then em. So `**bold *italic***` is valid; `*italic **bold***` round-trips to itself
- Literal asterisks in source text escape as `\*`, etc.
- Block-ID prefix (`^bk-XXXX `) stays at the start of each line, unchanged from M2

### 6.2 Parser extension

The existing `docFromMarkdown` matches block-level patterns (headings, scene break, dialogue, lists). The body-text of each block currently becomes a single text node. M2.5 extends it to tokenize the body for inline marks:

```ts
function markdownToInline(schema: Schema, line: string): Node[] {
  // Tokenize the line into a sequence of (text, marks) runs.
  // Recognized tokens:
  //   **...**     → strong open/close
  //   *...*       → em open/close (NOT inside a word: `f*oo*` parses literal)
  //   [text](url) → link
  //   \<x>        → literal <x>, where <x> ∈ { *, _, [, ], (, ), \ }
  // Disambiguation: an opening * must be preceded by whitespace, BOL, or
  // certain punctuation; a closing * must be followed by similar. This
  // matches CommonMark's "left-flanking" / "right-flanking" rule simplified.
  // For M2.5, we ship a pragmatic subset that handles 99% of literary use;
  // edge cases (snug-against-punctuation emphasis) fall through to literal.
  // Returns an array of schema.text(...) nodes with appropriate marks.
}
```

### 6.3 Round-trip property tests

A new `serialize.test.ts` adds:
- A handful of hand-written round-trip cases (each mark alone, marks combined, nested marks, escaped literals).
- A property test: generate random docs (paragraphs with random text + random mark spans), serialize, parse, assert equal to original. ~50 iterations.

The property test catches the easy regressions; the hand-written cases pin specific edge cases.

### 6.4 Backward compatibility with existing M2 scenes

Existing scene `.md` files contain no marks. They parse as plain text (no `*`, `[`, etc. to match) and load identically to M2 behavior. No migration needed; no schema version bump in SQLite.

## 7. Keyboard shortcuts (`app/src/editor/Editor.tsx`)

Three new entries in the editor's keymap, slotted between `history` and `baseKeymap` so they take precedence over baseKeymap defaults (which currently bind nothing to Mod-B/I/K):

```ts
keymap({
  "Mod-z": undo,
  "Mod-y": redo,
  "Mod-Shift-z": redo,
  "Mod-b": toggleMark(schema.marks.strong),
  "Mod-i": toggleMark(schema.marks.em),
  "Mod-k": openLinkPopupCommand,  // see § 9
  Enter: ...,                     // existing list-aware split
}),
```

`toggleMark` is the standard `prosemirror-commands` factory. The link command is custom because the popup is a React component that needs to know the current selection bounds + existing href (if any).

## 8. SelectionToolbar (`app/src/editor/SelectionToolbar.tsx`)

### 8.1 Mount + lifecycle

`<SelectionToolbar editorView={view} />` is mounted as a portal child of `<body>` from the `Editor` component (so it escapes the editor host's overflow clipping). The component:

1. Subscribes to ProseMirror selection changes via a small plugin that exposes `editorView.state.selection` reactively (or uses a React state mirror updated in `dispatchTransaction`).
2. When the selection is non-empty and the editor has focus, shows the toolbar.
3. Computes position via `view.coordsAtPos(selectionFrom)` + `view.coordsAtPos(selectionTo)`. Centers the toolbar horizontally between those two coords; positions it 8 px above the selection's top.
4. Listens for `Escape` keydown, `scroll` on the editor host, and `focusout` on the editor; any of these hides the toolbar.

### 8.2 Visual spec

- Container: pastel-glow capsule. `padding: 6px 10px`. `border-radius: var(--water-r-16)`. No border.
- Background: `color-mix(in oklch, var(--water-hue-coherence) 30%, var(--water-bg-paper))`.
- Outer glow: `box-shadow: 0 0 18px color-mix(in oklch, var(--water-hue-coherence) 50%, transparent)`.
- Layout: flex row, `gap: 6px`. Three icon buttons (Lucide `Bold`, `Italic`, `Link2`), 24 × 24 px clickable area, 16 px icon.
- Icon idle: `color: var(--water-fg-default)`, `background: transparent`.
- Icon active (mark fully covers selection): `background: color-mix(in oklch, var(--water-hue-flow) 40%, transparent)` — mint-cyan, the "current selection" cue.
- Fade-in: `opacity 0 → 1` over 240 ms, `transform: translateY(4px) → translateY(0)`, ease-out-soft.
- Fade-out: `opacity 1 → 0` over 120 ms, ease-in-out-water.
- Reduced-motion → 1 ms via existing tokens.

### 8.3 Click handlers

Clicking B / I dispatches `toggleMark(schema.marks.strong | em)` on the current view. Clicking Link opens the LinkPopup anchored just below the toolbar.

After click, focus stays on the editor (the toolbar doesn't steal focus; clicks use `mousedown`-event-prevention to avoid selection collapse).

## 9. LinkPopup (`app/src/editor/LinkPopup.tsx`)

### 9.1 Open paths

Three entry points:
1. Click Link icon in SelectionToolbar.
2. `Mod-K` with a non-empty selection (creates a link on the selected text).
3. `Mod-K` with a collapsed selection inside an existing link (edits that link).

For path 3, the popup pre-fills the input with the existing href and shows the Remove button.

### 9.2 Visual spec

- Same pastel-glow capsule style as the SelectionToolbar.
- Wider container (~280 px) to fit URL input.
- `<input type="url" placeholder="https://" autoFocus />` — borderless, transparent background, `var(--water-fg-default)` color.
- "Apply" button right-aligned. "Remove" button visible only in edit mode.
- Positioned 8 px below the toolbar (or below the selection, if the popup was triggered by Mod-K and the toolbar isn't visible).

### 9.3 URL validation

```ts
function isAcceptable(raw: string): boolean {
  const url = raw.trim();
  if (!url) return false;                                  // empty
  if (/^javascript:/i.test(url)) return false;             // XSS
  if (/^data:/i.test(url)) return false;                   // XSS
  return true;                                             // everything else OK
}
```

The `tauri-plugin-shell::open` call in § 10 also has its own scope guard (the capability config can restrict scheme), but we belt-and-suspenders the renderer too.

### 9.4 Close paths

- Apply with valid URL → mark applied, popup closes, focus returns to editor at original selection.
- Apply with empty input AND popup is in edit mode → equivalent to Remove.
- Remove (edit mode only) → link mark removed from the linked range, popup closes.
- Esc → popup closes without changes.
- Click outside → popup closes without changes.

## 10. Mod-click opens external (Tauri integration)

### 10.1 Rust side

Add `tauri-plugin-shell = "2"` to `app/src-tauri/Cargo.toml`. Register the plugin in `main.rs` via `.plugin(tauri_plugin_shell::init())`.

Add `"shell:default"` to `app/src-tauri/capabilities/default.json`. Restrict the `open` scope to `https`, `http`, and `mailto` URL schemes:

```json
{
  "identifier": "shell:allow-open",
  "allow": [
    { "url": "https://*" },
    { "url": "http://*" },
    { "url": "mailto:*" }
  ]
}
```

The capability scope is the second line of defense beyond the renderer's `javascript:` / `data:` guard.

### 10.2 Renderer side

Add to `app/src/ipc/commands.ts`:

```ts
openExternalLink: async (url: string): Promise<void> => {
  const { open } = await import("@tauri-apps/plugin-shell");
  await open(url);
},
```

In `Editor.tsx`'s `EditorView` props, add `handleClickOn`:

```ts
handleClickOn(view, _pos, node, _nodePos, event) {
  const isModClick = event.metaKey || event.ctrlKey;
  if (!isModClick) return false;
  const link = view.state.selection.$from.marks().find((m) => m.type.name === "link");
  if (!link?.attrs.href) return false;
  void ipc.openExternalLink(link.attrs.href as string);
  return true;  // tell PM we handled it
},
```

Rendered link elements get `title="Cmd-click to open"` (Mac) / `title="Ctrl-click to open"` (Win/Linux) via a small inline-mark renderer extension or a CSS `::after` hover state — we go with the title attribute for simplicity.

## 12. Plugin contract & extensibility

The mark-set is structured so future additions (strike, code, underline) are mechanical: add the mark spec to the schema, add the serializer/parser rules, add a shortcut + toolbar icon, add tests. Each new mark is ~40-60 LoC.

The SelectionToolbar's icon list is a static array; new marks slot in as new entries. The LinkPopup is link-specific and doesn't generalize.

## 14. Exit criteria

1. Select text in the editor, press `Mod-B` → text renders bold. Save scene; close + reopen project; bold persists. On-disk `.md` contains `**...**` at that span.
2. Same for `Mod-I` → `*...*`.
3. Select text, click Link icon (or `Mod-K`), enter URL, press Enter → text is linked. Hover shows URL via `title` attribute.
4. Mod-click (or Ctrl-click) on a rendered link → default browser opens the URL.
5. Selection toolbar fades in 240 ms on non-empty selection; fades out 120 ms on selection collapse, Esc, or blur. Reduced-motion mode: instant.
6. URL input rejects `javascript:foo` and `data:bar` without applying the mark.
7. Serializer property test: 50 randomly-marked docs round-trip identically.
8. All gates green:
   - `pnpm --filter @water/app test` → 88-90 passed (70 + ~18-20 new).

## 15. Risks & mitigations

| # | Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|---|
| 1 | CommonMark emphasis disambiguation edge cases (`a*b*c` snug-against-letter) | High | Low | Ship the pragmatic subset (strict left-/right-flanking simplified); document the small gap in spec § 6.2; property test catches major regressions |
| 2 | SelectionToolbar visually competes with pills when both visible | Low | Medium | Toolbar uses `--water-hue-coherence` (cool periwinkle); pills use speaker-specific hues; only ever one toolbar; auto-hides on blur |
| 3 | XSS via crafted URL | Low | High | Renderer regex guard rejects `javascript:`/`data:`; capability config restricts scheme; `tauri-plugin-shell::open` sandboxes |
| 4 | Empty-paragraph `:has()` placeholder rule interacts oddly with mark-only paragraphs | Low | Low | Add specific test: paragraph containing only mark delimiters with no text content still triggers placeholder. Existing rule already targets `:only-child:empty`, which an open-but-never-closed mark wouldn't satisfy |
| 5 | Portal-mounted toolbar bleeds outside Tauri webview window on edge selections | Low | Low | Clamp toolbar position to viewport on overflow; document with a small inline comment |
| 6 | `handleClickOn` Mod-click conflicts with PM's default selection behavior | Low | Low | Return `true` from handler to indicate consumption; PM doesn't fall through to selection extension |
