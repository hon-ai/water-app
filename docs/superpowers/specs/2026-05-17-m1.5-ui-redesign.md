# M1.5 — Visual Chrome Redesign Spec

**Status:** Approved
**Date:** 2026-05-17
**Base:** tag `m1.1` (commit `b1716de`)
**Successor:** M2 (Editor & Pill Engine) per `docs/superpowers/specs/2026-05-16-water-design.md` § 4.4

---

## 1. Background

M1 and M1.1 built a functionally complete writing app: scenes persist to disk in human-readable Markdown + TOML, the sidecar auto-spawns, snapshots fire on close, providers can be tested through the LLM router. But the visible surface is still the placeholder chrome that landed in M1's T38 — a cramped header with text tabs (`scenes diagnostics`), an inline project-creation form sitting in the viewport at all times, raw HTML inputs for the parent directory and project name, and three single-letter theme buttons (`L D A`).

M1.5 replaces that chrome with a sidebar-first, no-header layout in the spirit of OpenCode's clean monochrome aesthetic and Apple's writing apps (Bear, Ulysses, iA Writer). The pastel-glow design tokens from M1 T34 already encode the substrate; M1.5 puts them to work. **No new functionality is added** beyond what's needed to wire the chrome into the existing IPC surface, plus one small Rust addition (`SceneStore::rename`) so the new title field can save.

M1.5 is **not** the editor + pill engine. Those land in M2 on top of this chrome.

## 2. Hard scope

### 2.1 Phase A — Foundations

**A1. Renderer dependencies + font wiring.**
Add `lucide-react`, `@fontsource/ibm-plex-sans`, `@fontsource/ibm-plex-serif`, and `@tauri-apps/plugin-dialog` to `app/package.json`. Import the fontsource CSS via a new `app/src/styles/fonts.css` and reference IBM Plex through new `--water-font-sans` / `--water-font-serif` / `--water-font-mono` tokens in `tokens.css`. The `body` rule consumes `var(--water-font-sans)`.

**A2. Tauri dialog plugin (Rust side).**
Add `tauri-plugin-dialog` to `app/src-tauri/Cargo.toml`. Register the plugin in `main.rs` via `.plugin(tauri_plugin_dialog::init())`. Add `dialog:default` to `app/src-tauri/capabilities/default.json` so the renderer can invoke the native folder picker.

**A3. Typed IPC wrapper.**
Create `app/src/ipc/dialog.ts` exposing `dialog.pickFolder(): Promise<string | null>` that wraps `@tauri-apps/plugin-dialog`'s `open({ directory: true, multiple: false })`. One smoke test.

### 2.2 Phase B — Scope addition: `SceneStore::rename`

**B0. `SceneStore::rename` + `scene_rename` Tauri command + `ipc.sceneRename`.**
This is the one Rust addition. The Notion-style title (Q3.i — first line of editor IS the title in Plex Serif) requires a way to update a scene's name from the renderer. Today the IPC surface only has `scene_create`, `scene_read`, `scene_write_body`, `scene_list`. We add a fifth: `scene_rename(id, name)`.

- `crates/water-core/src/scene.rs`: new method `pub fn rename(&self, id: &Id, name: &str) -> Result<SceneRow>`. Reads the scene file, updates `frontmatter.name`, recomputes `updated_at`, writes back, re-hashes, then `UPDATE scene SET name = ?2, file_hash = ?3, updated_at = ?4 WHERE id = ?1`. Returns the refreshed `SceneRow`. Same shape as `write_body`.
- `app/src-tauri/src/commands/scene.rs`: new `#[tauri::command] scene_rename(state, id, name) -> Result<SceneInfo>`. Registers via `tauri::generate_handler!`. Same `(db, root)` clone-and-lock pattern as the other handlers.
- `app/src/ipc/commands.ts`: new `sceneRename(id, name): Promise<SceneInfo>` on the `ipc` object.
- One unit test in `scene.rs::tests` asserts rename round-trips name in both the DB and the frontmatter.

### 2.3 Phase B — Core chrome components

Each component is built in isolation with TDD-style smoke tests for click handlers + visible elements. Visual polish (specific colors, exact paddings) is not pixel-tested.

**B1. `IconRail` component.**
56 px wide. Vertical column. Lucide icons: `Droplet` (app mark, top), `FileText` (scenes nav), `User` (characters M3-stub), `Globe` (world M4-stub), bottom: `Settings` (gear). Active icon's container background = `--water-hue-flow` at 30 % alpha; inactive icons = `--water-fg-muted`; hover triggers native `[title]` tooltip with the icon's label. Props: `active: "scenes" | "characters" | "world"`, `onSelect(target)`, `onOpenSettings()`. The `User` and `Globe` icons fire `onSelect("characters")` / `onSelect("world")` but the parent App can no-op them in M1.5 (they become real surfaces in M3/M4).

**B2. `ScenesPanel` component.**
280 px wide, collapsible to 0 px (Q1.i — fully gone). Two parts:
- **Top**: project-name button. Click opens a dropdown menu (Q2.i) with three items: "Switch project…" (calls open-folder dialog), "Open folder…" (alias; same target as Switch), "Close project". The dropdown is a `<details>` element styled to look floating, OR a simple click-toggled div with click-outside dismissal. Uses Lucide `ChevronDown` as the drop indicator.
- **Body**: `+ new scene` button at top, then a vertical list of scenes. Each row shows scene name + word-count chip on the right. Active scene has `--water-hue-flow` at 30 % alpha background + a thin glow halo. The list is scrollable; no virtualization (M5 territory).

Collapse: a `ChevronLeft` button at the rail/panel boundary. Width is persisted via `localStorage("water:scenes-collapsed")`. When collapsed, the editor canvas claims the full remaining width.

**B3. `EditorCanvas` component.**
Replaces the M1.1 `PlaceholderEditor` directly in the App tree. Internally renders TWO stacked inputs visually fused into a single 720 px-max-width prose column:
- **Title input** (Plex Serif, `--water-fs-display`, no border, no outline). Bound to scene.name. Saves on blur via `ipc.sceneRename(sceneId, title)`. Refreshes `ScenesPanel` after success (App-level callback).
- **Body textarea** (Plex Sans, `--water-fs-body`, `--water-lh-body`, no border, no outline, no resize handle). Inherits the M1.1 dirty-flag + 2 s debounce autosave behavior, but moved into the canvas so the "saved at" chip can float top-right of the canvas (NOT below the textarea anymore).

The `PlaceholderEditor.tsx` from M1.1 is **deleted** and its dirty-flag logic is inlined into `EditorCanvas`. The two new tests from M1.1 T5 (`does NOT call scene_write_body when only the initial read…`, `calls scene_write_body 2s after the user types`) move to `EditorCanvas.test.tsx` and stay green.

The `^bk-XXXX` block-id tokens land at the end of paragraphs but are styled with `opacity: 0` (zero-width) so the writer sees clean prose.

**B4. `EmptyState` component.**
Shown when no project is open. Sits in the canvas area (the icon rail still renders, but the rail's nav icons are inactive). Two CTAs: "Create new project" (opens the CreateProjectSheet) and "Open existing" (opens the native folder dialog directly via `ipc.dialog.pickFolder` → `ipc.openProject`). Above the buttons: a large `Droplet` icon in `--water-hue-flow` + the word "Water" in Plex Serif `--water-fs-display`.

### 2.4 Phase C — Sheets

**C1. Sheet primitive + CreateProjectSheet.**
`app/src/sheets/Sheet.tsx`: a reusable slide-from-right modal primitive built on `<dialog>` (`HTMLDialogElement.showModal()` for native focus trap + ESC handling). Props: `open: boolean`, `onClose(): void`, `children`. Slide-in via CSS transform on `[data-state="open"]` (~280 ms `--water-ease-out-soft`).

`app/src/sheets/CreateProjectSheet.tsx`: two fields (project name + parent dir) and a "Browse…" button that fills the parent dir from `ipc.dialog.pickFolder`. Submit → `ipc.createProject(parent, name)` → `onCreated(info)` → sheet closes. Cancel button + ESC both `onClose()`.

**C2. SettingsSheet.**
`app/src/sheets/SettingsSheet.tsx`. Three sections, accordion-style:
- **Appearance**: a 3-state segmented control replacing the old `L D A` buttons. Uses `useTheme()` directly.
- **Providers**: list pulled from `diagnostics_status().provider_health`. Each row shows provider id, ok/fail chip, "Test" button (→ `ipc.providerTest`), "Set key…" button (opens an inline password input → `ipc.providerSetKey`).
- **Developer info**: collapsible `<details>` containing the current Diagnostics page's full JSON dump + sidecar + router rows. This replaces the standalone Diagnostics page entirely.

### 2.5 Phase D — Layout shell

**D1. Rewrite `App.tsx`.**
Compose the chrome:
```
<ThemeProvider>
  <div className="water-shell">
    <IconRail active={...} onSelect={...} onOpenSettings={...} />
    {!projectOpen
      ? <EmptyState onCreate={...} onOpen={...} />
      : <>
          <ScenesPanel ... />
          <EditorCanvas ... />
        </>}
    <CreateProjectSheet open={createOpen} onClose={...} onCreated={...} />
    <SettingsSheet open={settingsOpen} onClose={...} />
  </div>
</ThemeProvider>
```
The shell owns: project-open state (polled via `diagnostics_status` every 3 s, same as M1.1's Diagnostics page did), active scene id, sheet open/closed flags, sidebar collapsed flag. `App.test.tsx` updates to assert the icon rail's settings icon is reachable and the "Water" empty-state mark renders.

### 2.6 Phase E — Cleanup + acceptance

**E1. Delete legacy page files.**
Delete `app/src/pages/SceneList.tsx`, `app/src/pages/SceneList.test.tsx`, `app/src/pages/Diagnostics.tsx`. The whole `app/src/pages/` directory goes (empty after deletions). Verify no imports remain.

**E2. Acceptance checklist + tag `m1.5`.**
Update `docs/m1-acceptance-checklist.md` with the new chrome behaviors. Run the full quality-gate sweep (build, clippy, fmt, tests on both crates + renderer + sidecar pytest). Tag `m1.5`.

## 3. Out of scope (explicit)

These are tracked but explicitly not in M1.5:
- Block editor (rich-text, slash commands, block ids per spec § 5.3) — M2 replaces the body textarea with a real editor.
- Pill margin column + pill rendering — M2.
- Conversational Intake popup — M2.
- The Spatial Continuum zoom rail (zoom 0–3 levels) — M5.
- Custom title bar / frameless window — Tauri's default decorated window stays.
- Custom Water brand logo — Lucide `Droplet` is the placeholder; M7 polish replaces.
- Push-event subscription for `sidecar:status` — 3 s polling stays (good enough for the badge).
- Animations with spring physics — only the existing `--water-ease-*` / `--water-dur-*` tokens.
- Virtualization for long scene lists — M5 territory.
- Live `prefers-color-scheme` change handling — already a known follow-up from M1's T35.

## 4. Sequencing

Linear: A1 → A2 → A3 → B0 → B1 → B2 → B3 → B4 → C1 → C2 → D1 → E1 → E2.

A1/A2/A3 (foundations) must land before any chrome component. B0 (scene_rename) must land before B3 (EditorCanvas uses it). B1/B2/B3/B4 can in principle run in parallel but linear is simpler. D1 needs all the components in place. E1/E2 close out.

## 5. Risks

**R1. Tauri 2 dialog plugin permission syntax.** The `dialog:default` permission name is the Tauri 2.x convention but each minor version has tweaked the ACL names. Verified by building. If wrong, the build fails with a clear error and we switch to `dialog:allow-open`.

**R2. IBM Plex bundle weight.** Sans 400/500/600 + Serif 400/500 = ~400 KB of woff2. Acceptable for a desktop app. We don't subset in M1.5; M7 polish can.

**R3. `<dialog>` semantics in Tauri's webviews.** WebView2 (Windows) and WKWebView (macOS) both support `HTMLDialogElement.showModal()`. If we hit a quirk, we fall back to an absolutely-positioned div + manual ESC handler.

**R4. Scene rename + open file handle race.** `SceneStore::rename` writes the same `.md` file that `SceneStore::write_body` would. If a user is mid-typing when they blur the title input, we could race: write_body fires from the textarea debounce while rename is writing the frontmatter. M1 acceptable because both write the WHOLE file (read → mutate → write), so the last writer wins. Worst case a single keystroke goes missing. Solid fix is a per-scene write lock, deferred to M2.

**R5. Sidebar width persistence.** `localStorage` is per-origin; Tauri dev and packaged builds share the origin so persistence works in both.

## 7. After M1.5

Move to M2 (Editor & Pill Engine). The chrome built in M1.5 is the shell that M2's pill margin + block editor + Conversational Intake popup will plug into. The editor canvas's body textarea gets replaced by a real block editor with block IDs, pill anchoring, and the pastel-glow margin pill column. The Intake popup is a new sheet primitive variant (centered glowing card per spec § 5.3 instead of slide-from-right).
