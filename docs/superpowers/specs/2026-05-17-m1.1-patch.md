# M1.1 Foundation Patch — Spec

**Status:** Approved
**Date:** 2026-05-17
**Base:** tag `m1-foundation` (commit `8067a1b`)
**Successor:** M2 (Editor & Pill Engine, per `docs/superpowers/specs/2026-05-16-water-design.md` § 4.4)

---

## 1. Background

M1 shipped 40 tasks, 70 + 5 + 4 tests passing, all quality gates green. The final whole-implementation review surfaced one consistent pattern: **the Rust library code is solid and the per-task gates passed legitimately, but the Tauri-app glue layer is meaningfully incomplete.** Three major subsystems — snapshot scheduler, sidecar handle, sidecar supervisor — exist and are unit-tested but are never instantiated from the running product. Several smaller correctness bugs lurk behind happy-path tests.

M1.1 is the patch series that closes those gaps before M2 starts wiring the pill engine. **M2's first commit assumes the sidecar is auto-spawning and the router is actually populated**; M1.1 makes both true.

This is **not** a re-design. The UI redesign proposed in conversation on 2026-05-17 is explicitly parked.

## 2. Hard scope

### 2.1 Phase A — Correctness fixes

**A1. Fix `provider_test` router save.**
File: `app/src-tauri/src/commands/provider.rs:30-32`. Currently overwrites `state.router` with a `LlmRouter::new(vec![Arc::new(CannedProvider)])` regardless of which provider was tested. **Fix:** save a router built from the same `provider` Arc that was just exercised. **Acceptance:** after `provider_test("anthropic")` succeeds, `diagnostics_status().router_primary_id == Some("anthropic")`.

**A2. Fix `rebuild_from_truth` FK ordering.**
File: `crates/water-core/src/rebuild.rs`. Scenes currently INSERT with `pov_character_id` / `location_id` populated BEFORE characters and world entries exist. Any real project with character refs fails to rebuild. **Fix:** reorder `repopulate` to load characters and world before scenes. Also collect `scene_character_presence` rows during scene insertion and flush them after characters land (replacing the current silent-failure `INSERT OR IGNORE … .ok()` pattern). **Acceptance:** new integration test rebuilds a project whose single scene references one character + one world entry; verify row counts in scene, character, world_entry, scene_character_presence.

**A3. Fix `PlaceholderEditor` spurious-write bug.**
File: `app/src/components/PlaceholderEditor.tsx`. The 2 s autosave debounce fires on initial load because `ipc.sceneRead` → `setText(...)` triggers the write-effect's dependency. The just-read body is re-written, which churns `file_hash` and `updated_at` for every scene open. **Fix:** gate the write effect on a `dirty` flag set only inside `onChange`, reset to `false` after a successful save. **Acceptance:** opening a scene and immediately switching away within 5 s leaves `file_hash` and `updated_at` unchanged.

**A4. Surface malformed dev-keys.toml.**
File: `crates/water-core/src/llm/secrets.rs:20`. Current `toml::from_str(...).unwrap_or_default()` silently swallows parse errors, yielding confusing "no secret for provider" errors downstream. **Fix:** log via `tracing::warn!` when the file exists but fails to parse; still fall back to an empty map (preserve the "don't crash on a typo" invariant). **Acceptance:** unit test that writes garbage TOML to a temp `dev-keys.toml`, loads via `Secrets::load`, and asserts the warning is emitted (use `tracing-test` or a custom subscriber).

**A5. `.gitignore` cleanup.**
Add `*.water/` and `**/Test-Project.water/` to `.gitignore` so test projects created via the app during manual testing don't show as untracked. Remove the current stray `app/src-tauri/Test-Project.water/` directory from the working tree (it was created by the user during M1 manual acceptance). **Acceptance:** `git status` is clean after running through the M1 acceptance checklist.

### 2.2 Phase B — Wire library code into the Tauri app

**B1. Instantiate `SnapshotScheduler` on `open_project`.**
File: `app/src-tauri/src/commands/project.rs`, `app/src-tauri/src/state.rs`. Currently neither `open_project` nor `create_project` spawn the scheduler. **Fix:**
- Add `scheduler: Mutex<Option<SnapshotScheduler>>` to `OpenProject` (or as a sibling field on `AppState` — see B-meta below).
- On project open, call `SnapshotScheduler::spawn(db_arc, project_root)` and store the handle.
- On every `scene_create` (and possibly on `scene_list` first-load), call `scheduler.register(ActiveScene { scene_id, file_path })`.
- On `close_project`, call `scheduler.on_close().await` (fires `OnClose` snapshots for active scenes) then `scheduler.stop().await`.

**Acceptance:** a new integration test that:
1. Opens a project containing a scene with a non-trivial body.
2. Triggers `OnClose` via `close_project`.
3. Asserts at least one `OnClose`-trigger snapshot row exists in `snapshot` table for that scene.

**B-meta. Db sharing problem.** `SnapshotScheduler::spawn` takes `Arc<Mutex<Db>>` and the Tauri `AppState` holds `Mutex<Option<OpenProject>>` where `OpenProject.db: Db`. These can't share storage without restructuring. **Resolution:** move the DB into `Arc<Mutex<Db>>` at the `OpenProject` level — `OpenProject { root: PathBuf, db: Arc<Mutex<Db>>, ... }`. Commands that previously did `&project.db` now do `let db_guard = project.db.lock().await; let db = &*db_guard;`. This is a meaningful refactor; it must land before B1 or B2 will compile.

**B2. Instantiate `Sidecar` + `SidecarSupervisor` on `open_project`.**
File: `app/src-tauri/src/commands/project.rs`, plus new wiring in main.rs for tauri events. **Fix:**
- Add `sidecar: Option<Arc<Sidecar>>` and `supervisor: Option<SidecarSupervisor>` to `OpenProject` (or `AppState`).
- On `open_project`, locate `uv` via `which::which("uv")`, locate the sidecar workspace path (use `app/src-tauri/`'s parent for dev; resolve via `tauri::api::path::resource_dir` for packaged builds — verify the Tauri 2 API), pick a port (start at `18765` and increment until bind succeeds, or accept config), call `Sidecar::spawn(spec)`, then `SidecarSupervisor::spawn(sidecar.clone())`.
- Forward supervisor status events to the renderer via `tauri::Emitter::emit("sidecar:status", &status_event)`.
- On `close_project`, call `supervisor.stop()` then `sidecar.shutdown().await`.

**Acceptance:**
- Integration test: open a project → assert `diagnostics_status().sidecar.status == "ready"` within 8 s.
- Manual: relaunch the app, open a project, no need to run `uv run uvicorn` in a separate terminal — the sidecar is running.

**B3. Drain sidecar stdout/stderr.**
File: `crates/water-core/src/sidecar.rs:60-84`. After port discovery, the stdout `BufReader` is dropped and stderr is `Stdio::piped()` but never read. Pipes will fill (~64 KB on Windows) and uvicorn will block. **Fix:**
- After parsing the `WATER_SIDECAR_PORT` line, spawn a `tokio::task` that reads remaining stdout lines and `tracing::info!`s them with a `sidecar=stdout` field.
- Spawn a parallel `tokio::task` for stderr with `sidecar=stderr`.
- Both tasks terminate naturally when the child exits and the pipe closes.
- Add a unit test (using `tokio::process` against a small Bash/PowerShell helper script that prints to both pipes) verifying the tasks terminate cleanly. If a cross-platform helper is awkward, settle for a `#[ignore]`-d test against the real sidecar.

**Acceptance:** open a project, leave it for 10 minutes with the sidecar running, no deadlock or memory growth. (Manual; no automated test for the timing.)

**B4. Extend `diagnostics_status` with real fields.**
File: `app/src-tauri/src/commands/diagnostics.rs`, and the renderer's `app/src/pages/Diagnostics.tsx`. Current `DiagnosticsStatus { has_open_project, project_root, providers, router_configured }` is too thin. **Fix:** extend to include:

```rust
pub struct DiagnosticsStatus {
    pub has_open_project: bool,
    pub project_root: Option<String>,
    pub router_primary_id: Option<String>,
    pub sidecar: Option<SidecarInfo>,
    pub provider_health: Vec<ProviderHealth>,
}

pub struct SidecarInfo {
    pub base_url: String,
    pub status: String,        // "loading" | "ready" | "error"
    pub last_status_detail: Option<String>,
}

pub struct ProviderHealth {
    pub id: String,
    pub ok: bool,
    pub error: Option<String>,
}
```

`provider_health` calls `LlmRouter::health()` (already implemented at `router.rs:178`). `sidecar.status` reads the supervisor's current value via `supervisor.current()`. Update `Diagnostics.tsx` to render these in a structured layout (still JSON-dumped for the dev section, but with named sections for the new fields).

**Acceptance:** integration test asserting the shape; manual verification that the page renders the new fields after `open_project`.

### 2.3 Phase C — Production correctness

**C1. Tauri `capabilities/default.json`.**
File: `app/src-tauri/capabilities/default.json` (new). Tauri 2 requires per-window capability declarations for production builds. Currently `pnpm tauri build` would produce a binary that blocks all IPC commands. **Fix:**
```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "default",
  "description": "Default capability for the main window",
  "windows": ["main"],
  "permissions": [
    "core:default"
  ]
}
```
…and update `tauri.conf.json` to reference it if not auto-discovered. Verify by running `cargo tauri build --debug` (or the closest equivalent) and confirming the build succeeds.

**Acceptance:** `pnpm tauri build` (or `cargo tauri build`) completes without an error about missing capabilities. Manual launch of the produced binary shows the IPC commands work.

### 2.4 Phase D — Hygiene

**D1. `KNOWN_FRAGILE.md` entries 3–6.**
File: `KNOWN_FRAGILE.md`. Append four new entries:

3. **`serde_yaml` is unmaintained** (RUSTSEC-2024-0320). Surface is small (only scene frontmatter). Migrate to `serde_norway` or `serde_yml` if surface grows.
4. **`Secrets::load` warns-but-doesn't-fail on malformed dev-keys.toml** (after A4 fix). A user with a typo gets a `tracing::warn!` not a hard error.
5. **IPC schema drift between Rust `water_core::ipc` and Python `water_sidecar.routes.analyze`** is policed by one hand-pinned JSON literal in `ipc.rs::tests`. M2 should add a CI step that posts a known-bad payload to the running sidecar and asserts both ends reject it.
6. **`SnapshotScheduler` and `SidecarSupervisor` are restart-on-close-only.** M1.1 wires them into `open_project` lifecycle but does NOT add auto-restart on crash (the supervisor's 3-strike logic only breaks the watch task; it doesn't respawn). M2 can add real respawn logic if needed.

## 4. Sequencing

Phase A and Phase D can run in parallel. Phase B requires the `Db → Arc<Mutex<Db>>` refactor (B-meta) as a precondition; B1, B2, B3, B4 then proceed in any order though B2 depends on B-meta being done. Phase C is independent. Phase E is the final gate.

Suggested order: **B-meta → A1 → A2 → A3 → A4 → A5 → B3 → B1 → B2 → B4 → C1 → D1 → E1 → E2 → E3**.

(B3 before B1/B2 because the pipe drain is a precondition for any long-running sidecar.)

## 5. Risks

**R1. Db `Arc<Mutex>` refactor cascades through every Tauri command.** Every existing `&project.db` callsite needs to acquire the mutex. ~10 command handlers. Mechanical but tedious; risk is missing a callsite and getting a runtime deadlock if a guard isn't dropped before another command runs. Mitigation: scope each lock acquisition to the smallest block possible (`let row = { let db = project.db.lock().await; … };`).

**R2. Tauri 2 capability discovery on Windows.** The default discovery rules may need an explicit reference in `tauri.conf.json`. If `cargo tauri build` doesn't find `capabilities/default.json` automatically, we'll need to add `"capabilities": ["default"]` to `tauri.conf.json`'s `app.security` block. Cheap to discover and fix.

**R3. Sidecar port selection.** Hard-coding `18765` works for one open project but fails when two Water windows open simultaneously. M1.1 starts with hard-coded; if that breaks immediately we add `port: 0` discovery + bind-attempt-loop. Mitigation: pick port from an env var with a default, and document the limitation in KNOWN_FRAGILE entry 6.

**R4. Sidecar workspace path resolution.** In dev mode the path is `<repo>/sidecar/` relative to the Tauri working directory. In a packaged build it must be a bundled resource. Tauri 2's `tauri::api::path::resource_dir()` is the official answer but the API surface differs across Tauri 2 versions. Mitigation: implement the dev-mode path first (which is what we need for M1.1 acceptance), document the packaged-build resolution as a future task. The packaged build doesn't currently work for other reasons (capabilities, code signing) so deferring is honest.
