## 1. The one-paragraph product context

Water is a Tauri 2 desktop writing app with one heretical UX rule: **the writer never types into a chat box and the LLM never sees a chat box.** M1–M4 shipped the pill engine + scene/character/world bibles + the world-drift cartographer. M4.5 was a UX polish sprint (cool-gray light substrate, elevation tokens, sheet polish, intake/settings polish, navigation discoverability). M5 ships the Heatmap audiovisualizer — a 24 px ambient barometer above each scene's editor canvas showing pacing/valence/coherence/presence/world-refs. M6 takes the macro step up: a spatial canvas across many scenes, where the writer can see the project as a whole — drag scenes around, group them, see heat strips line up across the act/sequence/chapter structure.

## 3. M6 scope (Macro Spatial Scene Canvas)

Locked design spec to be authored as the first task. Locked v1 design spec is `docs/superpowers/specs/2026-05-16-water-design.md` § 4.8 (M6) and § 3.9 (spatial canvas schema).

**M6 ships:**

1. **A new "Canvas" nav target** on the IconRail — a third-tier surface alongside Scenes / Characters / World.
2. **Free-form scene placement** — drag scene cards on a 2D canvas. Position persists in a new `scene.canvas_x` / `canvas_y` column pair.
3. **Per-scene heat sparklines** — each scene card on the canvas shows a mini version of its heat strip so the writer can see the shape of the project at a glance.
4. **Optional grouping** — pinch / lasso a group of scenes to assign a label ("Act II", "the cove sequence"). Group is purely a visual hint, not a model entity in v1.
5. **Reading order overlay** — toggle to draw arrows between scenes in `manuscript.scenes.ordering`. Lets the writer compare spatial layout vs. narrative order.

**Exit criteria (sketch — refine in the M6 spec):**

- Canvas renders all scenes in the active manuscript at their persisted positions.
- Dragging a scene persists position via debounced IPC.
- Each scene card shows a 60 px-wide heat sparkline derived from `HeatStore` data.
- Reading-order overlay toggles on/off without page reload.

**Explicitly NOT in M6:**

- Multi-manuscript / multi-project canvases.
- Auto-layout heuristics ("snap to act boundaries").
- Macro-level heatmap aggregation across scenes (M7 candidate).

## 4. Constraints carried forward (M1–M5)

All M2/M3/M4/M5 hard constraints still apply. M6-specific additions:

1. **The Canvas surface is observational, not generative.** Like the Heatmap strip, the macro view shows the writer where they are. No LLM ever sees the canvas state directly.
2. **Spatial position is not narrative position.** Manuscript ordering remains the authoritative reading order. Canvas position is a private writer-tool — the writer can move scenes around without changing what gets read first.
3. **No new providers / cloud deps.**
4. **Lock ordering invariant** (KNOWN_FRAGILE #6): `project lock → drop → write-lock → db lock`. M6's `scene_canvas_set_position` follows the same rule via the existing `SceneWriteLocks`.

## 7. Open questions to bring into the M6 brainstorm

1. **Canvas projection** — fixed-size canvas with pan/zoom (Figma-style) vs. infinite scroll with floating cards (Miro-style)? Tradeoffs around discoverability + spatial intuition.
2. **Default layout** — when a writer first opens Canvas, do scenes auto-arrange in a row by manuscript order, or are they all at (0,0) waiting to be placed?
3. **Heat sparkline accuracy at small sizes** — 60 px wide × 8 px tall has only ~10 pixels per paragraph for a 5-paragraph scene. Aggregate by scene-mean? Show only the dominant metric?
4. **Group semantics** — pure visual hint vs. real model entity (own table, own writer-facing label, shows up in scene rename)?
5. **Reading-order overlay** — straight arrows scene-to-scene, or smoothed curves (avoid criss-cross when the writer's spatial layout doesn't match manuscript order)?
