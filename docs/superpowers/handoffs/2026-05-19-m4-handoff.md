## 1. The one-paragraph product context

Water is a Tauri 2 desktop writing app with one heretical UX rule: **the writer never types into a chat box and the LLM never sees a chat box.** The LLM sees only the manuscript text plus the click history of small "pills" that surface in the margin while the writer works. Pills come from named "speakers" — generic personas (Echo / Architect / Editor / Cartographer / Chorus, shipped M2) or character voices drawn from LSM v2.1 sheets (M3) — and their language is reactive/observational, never instructive. The writer can expand a pill into a bouquet of three child pills, expand any child into another bouquet, regenerate a bouquet, pin a pill to a quiet column at the right margin, or dismiss the whole thread. Local-first with cloud opt-in.

**M3 shipped character voices.** M4 wires world entries into the same engine so the Cartographer persona has standing and `world_drift` can fire.

## 3. The M4 spec, in your hands

The locked v1 design spec is `docs/superpowers/specs/2026-05-16-water-design.md`. **Read § 3.6 (World templates), § 4.6 (M4 plan), § 3.0 schema (`world_segment` + `world_entry` tables already exist).** No separate M4 spec doc yet — write one as your first phase (mirrors how M3 spec was written before the M3 plan).

**M4 ships:**

1. **6 default world segments** — `concept` (single doc), `locations` (collection), `politics_and_social` (single doc), `culture` (single doc), `world` (single doc), `history` (single doc). Schemas in § 3.6.
2. **Segment editor** — single-doc OR collection mode driven by `is_collection`. Collection entries get their own intake; single-doc segments get one inline-editable sheet.
3. **World index view** — soft glowing tile per segment + collection items. Same visual language as CharacterIndex.
4. **Segment template editor** (settings panel) — add/remove fields, add new segments. v1 restriction: built-in segments are hidden-not-removed.
5. **`world_drift` trigger** — cheap exact-match check between named entities in scene text and `world_entry.name` + aliases; if mismatch found, LLM follow-up via the M3 two-stage confirmation pattern; fires Cartographer.
6. **Character voice prompt enrichment** — voice template can pull `world_entry` excerpts (e.g. scene's `location_id`'s `sensory_detail`). This is the cross-milestone integration: M3's `CharacterSpeaker::from_row` gains access to a `WorldRegistry` (parallel to `CharacterRegistry`).

**Exit criteria** (verbatim from spec § 4.6):

- All built-in segments work with intake.
- User can add a new segment with a custom template and intake walks it.
- Character voice prompts can pull relevant `world_entry` excerpts.
- `world_drift` correctly identifies a planted contradiction in test scenes.

**Explicitly NOT in M4:**

- Heatmap audiovisualizer (M5)
- Macro spatial scene canvas (M6)
- Customization, polish, tester build (M7)
- Embedding-based `world_drift` upgrade (KNOWN_FRAGILE escape hatch lives in M5+)

## 4. The absolutely-do-not-violate constraints

All M2/M3 hard constraints still apply. M4-specific additions:

1. **The LLM NEVER receives conversational input from the user.** This applies during world intake too — the writer's world-segment answers ARE the manuscript-adjacent context. No chat textbox.
2. **Reactive / observational pill tone.** Cartographer pills follow the same blacklist (`you should / consider / try / I think you / as an AI`) at the three layers (prompt clause, PASS, post-hoc regex).
3. **Universe-first, personas-second.** With M4, when a scene mentions a `world_entry` (via `location_id` or name-match), the orchestrator should prefer Cartographer over Echo/Architect for relevant triggers. Voice router's `Either` track gains a Cartographer-prefer rule analogous to M3's POV-prefer.
4. **Local-first.** No new cloud deps.
5. **Determinism.** `(trigger, scene_state, project_state, persona_state, character_state, world_state)` → same prompt assembly + speaker selection.
6. **Human-readable on disk.** `world/<segment>/<entry-ulid>.toml` for collections; `world/<segment>.toml` for single-doc segments. Same contract as `scenes/` and `characters/`. SQLite is rebuildable from disk per § 3.8.
7. **Plugin-contract design.** Whatever M4 adds (likely a `WorldSpeaker` for the Cartographer-with-world-context path) is a `Speaker` impl alongside `CharacterSpeaker` / `PersonaSpeaker`.
8. **Pastel-glow visual identity + Notion/Apple-fluid motion.** Reuse the existing tokens; segment tiles use the same hue token system characters use (round-robin per `world_segment`).
9. **`ConversationalIntake` is schema-agnostic and MUST stay that way.** The M3 component takes `schema: IntakeSchemaSection[]` of arbitrary shape. Do NOT hardcode LSM-specific anything in M4 work that touches `app/src/intake/`. World schemas should be representable as `IntakeSchemaSection[]`.

### Step 1: Decide M3-polish vs. M4 ordering (REQUIRED)

The M3-polish backlog (Section 6) has ~17 items. Two reasonable orderings:

- **A — Polish-first sprint:** spend ~half a day clearing the M3 backlog before starting M4. Pros: clean state for M4, no drift on issues that touched the same files. Cons: delays M4 momentum.
- **B — M4-first, polish opportunistically:** start M4 brainstorming immediately; clear M3 polish items WHEN M4 work touches the same module (e.g. T2 of M4 will likely touch migrations, so do the M3 migration-test polish then).

Default to B unless the user explicitly asks for A. The M3 polish items are all non-blocking and the WorldRegistry/WorldStore work will naturally re-enter the modules where most polish hotspots live.

### Step 2: Brainstorm session (REQUIRED — use the skill)

Invoke `superpowers:brainstorming`. Product questions to surface:

- **Segment template descriptor format** — JSON-Schema-shaped or LSM-style `IntakeField[]` (M3 chose the latter)? If `IntakeField[]`, the segment editor is "edit a list of IntakeField records" — solvable with the same primitives.
- **Collection segment intake** — when adding a new entry to `locations`, walks the template like character intake. When viewing a collection, it's a CharacterIndex-shaped grid.
- **`is_collection` UX distinction** — single-doc segments show one sheet; collection segments show an index grid + per-entry sheets. Different routing inside `WorldsSurface`.
- **`world_drift` heuristic** — name-string-matching is the cheapest stage 1 (mirrors M3 autosuggest). Stage 2 LLM confirmation when a scene mentions a `world_entry.name` whose attributes contradict the scene context. KNOWN_FRAGILE inheritance: the same English-only tokenization limitation applies (#17).
- **Cartographer trigger conditions** — when does the Cartographer have priority over other personas? When a scene's `location_id` is set and the recent paragraph contains text related to that location's attributes. Voice router `Either` track gains a Cartographer rule.
- **World template editor scope** — full settings UI? Or v1 a config-file-only path? The spec implies a UI but doesn't mandate a polished one. v1-minimum: add fields via TOML edit; full UI deferred to M7 polish.
- **Naming conflicts (cross-character vs. cross-world)** — `Aren` could be a character AND a place. Autosuggest needs to disambiguate. Spec implicit: prefer character match over world match when the scene also lists the character in `characters_present`.

## 7. Hard-earned lessons from M3 (carry these forward)

### Process

- **Plan code is always stale by the time the implementer reads it.** EVERY M3 task required adaptations. The pattern that worked: spell out adaptations explicitly in the implementer's prompt rather than letting them re-derive them. Maintain an "adaptations playbook" (see § 7 of the M3 plan).
- **Combined single-reviewer for narrow glue tasks beats dual-review.** T16, T17, T19, T20, T21 used one thorough reviewer (or focused spot-check) instead of two parallel subagents. Quality matched dual-review at half the context cost. Reserve dual-review for the heavyweight phase-opening tasks (T1, T7, T11, T15, T18).
- **"Surface only blockers" cadence is correct.** Important findings go into a deferred-fixes batch; Critical findings stop the task and get fixed inline. The cadence saved roughly 30% of context vs. fix-every-finding-immediately.
- **Two Critical bugs slipped into T15** (autoFocus lost across same-kind transitions; `onComplete` never fires when pre-complete). Both were latent — green tests passed for the WRONG reasons. The lesson: when the implementer's tests are passing but the task involves keyboard focus or mount-time effects, EXPLICITLY ask for a regression test that covers the negative case. Subsequent tasks (T16-T21) didn't repeat this class of bug because the adaptations playbook now flags it.
- **Reviewer suggestions are not always correct.** T16's reviewer suggested a 2-line fix (`const writingFor = characterId; ... if (writingFor !== characterId)`) that had a closure-capture bug. Verifying the fix actually works before applying is essential.

### Vitest / TypeScript gotchas (M3-specific)

- **`#[serde(flatten)]` Rust types deserialize with section keys at TOP LEVEL on the TS side.** `CharacterFile` shape is `{id, name, schema_version, [key]: unknown}` — there is NO `data` wrapper. Plan documents that assumed `file.data.main.full_name` were uniformly WRONG; reality is `file.main.full_name`. Likely repeats for `WorldEntry`.
- **`IntakeField.id` is a dotted path** (`"main.full_name"`), not a leaf name. The `ConversationalIntake` component looks up `values[field.id]` so the flatten helper must produce dotted-path keys.
- **`IntakeFieldKind` is a discriminated union** `{type: "short_text"} | {type: "long_text"} | {type: "string_list"} | {type: "choice", options: string[]}`. Match on `kind.type`, not on a flat string. `optional_skip: boolean` (NOT `required: boolean` — semantics inverted).
- **All IPC lives on the `ipc{}` singleton in `app/src/ipc/commands.ts`.** There is no `../ipc/character`, no `../ipc/scene`. Plan-writers should reference `ipc.X` consistently.
- **`key={field.id}` on the active input** is necessary across same-kind field transitions or `autoFocus` silently no-ops (T15 lesson). Pattern: any component that conditionally renders the same element type across state changes needs an explicit `key`.
- **Mount-time `useEffect` for `onComplete` if all-fields-already-answered.** ConversationalIntake T15 lesson. Pattern: any "completion stub" branch should fire its completion callback from a mount-only `useEffect`, not just render text.
- **Write-path race guard requires `useRef`, not closure-captured prop comparison.** T16 lesson — closure captures both `writingFor` and the comparison target from the same prop value. Use `const idRef = useRef(prop); idRef.current = prop;` and compare `writingFor !== idRef.current` post-await.
- **`flattenCharacterData.ts` extracted to `app/src/characters/`** is the shared helper. Refactor it to a generic `flattenSerdeFlatten.ts` under `app/src/util/` when M4 hits a second consumer (WorldEntry).
- **Mock `vi.mock("../ipc/commands")` to replace the `ipc` singleton.** Factory must be hoist-safe (no outer references). See `CharacterIntakeSheet.test.tsx` for the canonical pattern.

### Rust gotchas (M3-specific, carrying forward from M2)

- **`_core` async fn pattern is now required for every Tauri command.** `tauri::State<'_, T>` has no test constructor; the `_core` fn takes unpacked deps directly so tests can drive it. Every M3 command (T12-T14, T21) uses this. Bake it into the M4 plan.
- **Lock ordering invariant (KNOWN_FRAGILE #6):** `project lock → drop → scene-write-lock OR character-write-lock → db lock`. M4 will introduce `world_write_locks: WorldWriteLocks` (per-entry id-keyed DashMap), parallel to `SceneWriteLocks` + `CharacterWriteLocks`. Same ordering: world-write-lock before db-lock.
- **`Db: Send + !Sync`** (rusqlite internal RefCell). Wrap in `Arc<Mutex<Db>>`.
- **`rusqlite::Statement: !Send`** — never hold across `.await`. Drain into owned types inside a sync block.
- **`unchecked_transaction()`** is the documented escape hatch when only `&Connection` is available (T13 uses it for the link/unlink+POV transactional cascade).
- **Migrations are at v3** post-M3 (added `character.hue_token` in T2). M4 will likely be v4 (`world_segment.hue_token` if collection segments get per-entry hue cycling).
- **`SecondsSinceEpoch` / `created_at`** uses `i64` Unix seconds throughout. Same pattern for `world_entry.created_at`.

### Windows / PowerShell

- **PATH setup at every fresh shell.** `$env:Path = "$env:USERPROFILE\.cargo\bin;$env:Path"`.
- **`Set-Content -Encoding UTF8` writes a BOM** in PowerShell 5.1. Use `[System.IO.File]::WriteAllText(...)` for clean UTF-8 fixtures.
- **CRLF/LF git warnings** are environmental. Ignore.
- **`NativeCommandError` on cargo stderr** is cosmetic. Ignore.
- **`Get-Content` and `Read` tool disagree on em-dash encoding.** Use the Read tool's output when constructing `oldString` for an `edit` operation; PowerShell's `Get-Content` may show `�?"` for what's actually `—`.
