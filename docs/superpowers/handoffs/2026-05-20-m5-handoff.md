## 1. The one-paragraph product context

Water is a Tauri 2 desktop writing app with one heretical UX rule: **the writer never types into a chat box and the LLM never sees a chat box.** The LLM sees only the manuscript text plus the click history of small "pills" that surface in the margin while the writer works. Pills come from named "speakers" — generic personas (Echo / Architect / Editor / Cartographer / Chorus) or character voices drawn from LSM v2.1 sheets — and their language is reactive/observational, never instructive. The writer can expand a pill into a bouquet of three child pills, expand any child into another bouquet, regenerate a bouquet, pin a pill to a quiet column at the right margin, or dismiss the whole thread. Local-first with cloud opt-in.

**M4 shipped the World/Setting Bible.** M5 will ship the Heatmap audiovisualizer that gives the writer a per-scene visualization of pacing, valence, coherence, character presence, and now world-entry references.

## 3. The M5 scope, in your hands

The locked v1 design spec is `docs/superpowers/specs/2026-05-16-water-design.md`. **Read § 4.7 (M5 plan) + § 3.7 (Heatmap schema) + § 6.4 (heat metrics) before writing the M5 spec.** No separate M5 spec doc yet — write one as your first phase (same pattern that produced M3 and M4 specs).

**M5 ships:**

1. **Heatmap visualization surface.** Per-scene strip rendering pacing, valence, coherence, character-presence intensity, and world-entry references. Reactively updates as the writer types.
2. **Heat metric computation.** Per-paragraph rolling scores; cached in SQLite, recomputed on body change with O(Δ) cost.
3. **Heatmap-aware pill triggers.** A few existing triggers (pace_floor, scene_flow_dip) gain heatmap awareness — they fire on heatmap-derived signals rather than ad-hoc heuristics.
4. **Optional: embedding upgrade path.** Spec § 4.7 mentions M5 as the natural place to land embedding-based versions of the M2/M3 cheap heuristics (anti-loop Jaccard, lemma overlap, contextual-overlap). Whether this lands in M5 or M6 depends on scope.

**Exit criteria** (verbatim from spec § 4.7):

- Heatmap strip renders all five metrics for the active scene.
- Writer can drag the heatmap to navigate the scene.
- Heatmap recomputes within 1 s of an autosave landing on a 5 000-word scene.
- At least one pill trigger has been migrated to a heatmap-derived signal.

**Explicitly NOT in M5:**

- Macro spatial scene canvas (M6).
- Customization, polish, tester build (M7).
- Multi-scene heatmap aggregation (M6 candidate).

## 4. The absolutely-do-not-violate constraints

All M2/M3/M4 hard constraints still apply. M5-specific additions will surface once you write the M5 spec; until then, inherit the M4 set:

1. **The LLM NEVER receives conversational input from the user.**
2. **Reactive / observational pill tone.** Persona blacklist + 3-layer enforcement (prompt clause, PASS, post-hoc regex).
3. **Local-first.** No new cloud deps.
4. **Determinism.** `(trigger, scene_state, project_state, persona_state, character_state, world_state)` → same prompt assembly + speaker selection.
5. **Human-readable on disk.** `world/<segment>/<entry-ulid>.toml` for collections; `world/<segment>.toml` for single-doc. SQLite is rebuildable from disk per § 3.8.
6. **Plugin-contract design.** Whatever M5 adds should remain `Speaker`-compatible / `Trigger`-compatible.
7. **Pastel-glow visual identity + Notion/Apple-fluid motion.**
8. **`ConversationalIntake` MUST stay schema-agnostic.**
9. **Lock ordering invariant** (KNOWN_FRAGILE #6): `project lock → drop → write-lock → db lock`.

## 7. M4 polish carry-forward (new debt from this milestone)

These are issues I noticed during implementation but did not address. None are blocking. Roll into a polish PR or opportunistically during M5 work when M5 work touches the same module.

1. **`SceneContext::empty` in `voice/mod.rs`** generates a fresh `Id::new()` per call. Cheap (ULID), but it's a small amount of nondeterminism that shows up in registry-build paths. M5 polish: have `empty()` return a static `Id` (or `Id::nil()` if such a thing is added).
2. **`pill_pin_core` builds a `WorldStore` inside the closure** that holds the DB lock. The DB lock is `Mutex<Db>`; the store borrows `&Db`. This is correct but somewhat tight — if the seed write ever stalls, the pin path blocks. Consider a separate write-lock for the stub-creation branch.
3. **`Bouquet.tsx` dispatches `water:nav-world-entry` via `window`** — works, but couples Bouquet to the worlds-surface routing primitive via a string event name. M5 may want a typed event bus.
4. **`SegmentTemplateEditor` doesn't normalize repeated whitespace in derived field ids.** "System  Name" (two spaces) becomes `main.system_name` after the `[^a-z0-9]+` collapse; "System Name " (trailing space) likewise. Fine for v1; future-proof by surfacing the derived id under the input.
5. **`WorldSettingsMenu` confirms via `window.confirm`** — synchronous, blocks the UI thread, and is jarring against the pastel-glow chrome. Spec § 4.7 doesn't require a custom modal, but it's worth replacing with a `<Sheet>`-styled confirm in M7 polish.
6. **The `pinned_pill.origin_trigger` column duplicates `trigger_class`.** Both store the same value today (`pill.trigger_id`). The split exists so M5+'s Heatmap can attach richer classification metadata without disturbing the origin-trigger lookup used by the Chorus-stub branch. If M5 doesn't actually need the split, drop `origin_trigger` in a v5 migration.

## 7. Hard-earned lessons from M4 (carry these forward)

### Process

- **TDD discipline held for ~70% of M4 tasks.** Tasks 25, 31, 32 followed strict RED→GREEN; task 29 wrote test + implementation together (still verified the implementation against behavioral assertions, but the "watch it fail first" step was skipped). When the test scaffolding requires new types or modules that don't exist yet, the RED step degenerates into a compile-error — not a behavioral failure. Pre-build the type signature with a `todo!()` body to get a meaningful RED.
- **Migrations + schema mirror tests are cheap insurance.** The v4 migration tests caught two typos before any other code ran. Replicate the M3 T2 / M4 T1 pattern for M5's heatmap tables.
- **`_core` async-fn extraction paid off again.** Every M4 Tauri command has a testable `*_core`. The pill-pin tests (T29) drove the stub-creation branch via the core fn alone; no Tauri harness required. Bake this pattern into M5 from the start.
- **Mocking only the IPCs you exercise leaves landmines.** WorldsSurface.test.tsx's stub-routing test failed once because the rendered WorldEntrySheet called an IPC the test hadn't mocked. The fix was to mock the full set the downstream component touches. M5: when adding a new route, list every IPC the destination component reads on mount.

### TypeScript / Vitest

- **`pillPin` returning `Promise<void>` vs. `Promise<PinPillResponse>`** is a silent type-narrowing migration. Existing `void ipc.pillPin(...)` call sites compile under both shapes (the `void` operator discards). Tests that mocked `pillPin: vi.fn().mockResolvedValue(undefined)` continue to pass at runtime but are wrong at the type level — `.mockResolvedValue` in `vi.fn()` is typed as `any`, so the mismatch never surfaces. M5: when widening a return type, grep mock factories and update them too.
- **`findByRole("button", { name: /Foo/ })`** is ambiguous when both a primary button and a `Dismiss Foo` button are in scope. The fix is a precise regex anchor (`/^Foo$/`) or `findByRole("button", { name: "Foo" })` (exact match). T28 ate ~20 minutes on this.
- **`findByText` is also ambiguous when an `<option>` and a `<button>` share the same text content.** Use a more specific selector (role + name) to disambiguate.
- **`findByText`/`findByLabelText` race against React effects.** When the assertion races a `useEffect` that fires an async IPC, prefer `findByText` (which waits) over `getByText` (which doesn't).

### Rust

- **`Db: Send + !Sync`** still bites in async paths. M4 introduced no new lock-ordering surprises, but the pattern `lock → drop → lock again` reappears in `pill_pin_core` (two separate `db.lock().await` calls bracketing the stub-create branch). The plan was to do this in one critical section, but the WorldStore needs `&Db` so the lock has to land once, and the test fixture wants to inspect the post-write state, which requires a fresh lock. Working as intended; flagged for awareness.
- **`rusqlite::Statement: !Send`** rules still apply. Drain queries to owned `Vec<T>` inside a sync block before yielding.

### Windows / PowerShell

- **`[System.IO.File]::WriteAllText(...)` for clean UTF-8** is still the safest path. PowerShell 5.1's `Set-Content -Encoding UTF8` writes a BOM, which `toml::from_str` accepts but `serde_json::from_str` does not.
- **CRLF/LF warnings on every commit** are environmental. Ignore.
- **`NativeCommandError` on cargo stderr** is cosmetic; the exit code is what matters.
